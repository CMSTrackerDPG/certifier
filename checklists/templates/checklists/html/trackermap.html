It shows the status of every modules, classified by partition and layer(disk). Tracker Maps are used for two different purpose. First, the scrutiny of the QTestAlarm maps and bad componenets allow to spot large problems. Second, all other features of the tool can help to investigate specific problems observed during data taking.
This page contains the list of checks to be done for every run, but also some useful information about possible issues and links to the documentation.

Tracker Map Check List

First, make sure the tracker maps are indeed produced. Then, in order to ensure that no big problems are ongoing, try to answer the following questions, looking at the QTestAlarm map :
  • Are there a large number of modues with high number of digis and/or clusters ?
  • Are there new bad modules concentrated to the same location in the map ? Or bad modules correlated, such as connected to the same power group, or "align" along the same direction (for example, two overlaping area between two layers/disks) ?
  • Is there big difference in comparison with the reference ? Please note anything different and try to understand these differences.
  • Check the number if bad modules. Is it very different from the reference run ?

Tips for investigating potential issues

  • First things to do : read the elog and search for anything unusal. Any problems observed during operations at P5 should be mention there, don't forget to look at the Central Shift Leader too
  • if you identify large dead regions, check excluded FEDs or large PS group problems
  • if you observed a larger number of modules with high number of clusters or digis, spread a bit randomly in a partition, check for recent upload of new calibrations, make sure the S/N distributions (in the GUI) is similar to the reference,
  • if you observed of FED errors, search for power group or DCS issues ; they might show up as FED errors,
  • if you see noisy modules (large number if clusters and digis), identify them in the list of bad modules, check their location using the cabling web interface link and check if they are in the list of known noisy modules on the whiteboard.
